# 09. 디스코드 채널/봇 세팅 가이드 (실전)

## 왜 이 문서가 필요한가
처음에 봇을 여러 개 붙이면 **중복응답/멘션 혼선**이 거의 반드시 생깁니다.
실전 권장값은:
- 인입 봇은 최대한 1개(또는 역할 분리된 2개)
- 채널별 역할을 고정
- 로그 전용 채널 분리

---

## 1) 채널 역할 분리(권장)
- deep-work 채널: 구현/설계/긴 대화
- ops/status 채널: 상태/에러/진행률 요약
- study 채널: 학습/논문/질문
- diary 채널: 회고/일과정리

핵심: 채널별 목적이 겹치지 않게 설계.

---

## 2) 봇 계정 설계
### 최소 구성(초보 추천)
- **1개 봇으로 인입/응답 통합**
- 로그는 같은 봇이 logger 채널로만 추가 전송

### 확장 구성(운영 안정화)
- `caine` : 실행/코딩 응답
- `bubbles` : 상태/알림
- `gigachad` : 학습/회고

주의: 같은 채널에서 여러 계정이 동시에 응답하면 중복응답 위험이 커짐.

---

## 3) 멘션 정책
- 기본: requireMention=true
- 예외 채널(deep-work/ops/study/diary): requireMention=false
- 계정별 오버라이드로 잡음 제어

실무 팁:
- `allowBots=true`면 봇 메시지 재섭취 가능성 점검 필수.
- 중복응답이 보이면 mention 패턴부터 좁혀라.

---

## 4) 로그 봇 세팅법
로그봇 역할은 "대화 참여"가 아니라 "운영 가시성"입니다.

필수 규칙:
1. logger 채널 고정
2. 중복 억제(fingerprint) 적용
3. critical만 main 채널 에스컬레이션

예시 이벤트:
- [NEW] 새 서브에이전트 시작
- [UPDATE] 상태 전환
- [BLOCKED]/[FAILED] 에러
- [DONE] 완료

---

## 5) 문제 재발 방지 체크리스트
- [ ] 채널-역할 맵 문서화
- [ ] 인입 바인딩 중복 여부 점검
- [ ] 멘션 규칙 채널/계정별 검증
- [ ] 중복응답 테스트(같은 질문 3회)
- [ ] logger/main 분리 작동 확인

---

## 6) 네가 실제 겪은 시행착오에서 나온 결론
- "봇 많으면 편하겠지" → 초기엔 오히려 혼선 증가
- "로그를 메인 채널에 계속 뿌리기" → 노이즈 폭증
- 결론: **인입 단순화 + 로그 분리 + 이벤트 기반**이 정답
